# Plan

El objetivo debe ser mover este fork hacia un modelo de mantenimiento donde `upstream/main` siga siendo la base funcional y el valor propio del fork quede concentrado en una capa pequeña y explícita: el bridge HTTP local para Every Code y, solo si sigue justificándose, ajustes mínimos de selección/modelo. Hoy esa separación todavía no existe; el historial de chat quedó entretejido en navegación, DI, settings y viewmodels, y eso eleva mucho el costo de seguir integrando cambios upstream.

## Requirements
- Conservar y endurecer la API HTTP local en `127.0.0.1` como feature principal del fork.
- Seguir integrando releases recientes de `google-ai-edge/gallery` sin asumir merges limpios.
- Reducir o retirar el subsistema de chat history si complica la convergencia con upstream.
- Mantener Tiny Garden restaurado y evitar que vuelva a divergir innecesariamente.
- Trabajar de forma segura sobre árbol sucio, con checkpoints frecuentes y rollback fácil.

## Scope
- In:
  - Integración selectiva de features recientes de upstream en chat UX, configuración, importación/modelos y tareas restauradas.
  - Refactor de la personalización local HTTP para aislarla.
  - De-integración limpia del chat history persistente.
  - Estrategia de sync con upstream.
- Out:
  - Reescritura arquitectónica grande.
  - Nuevas features locales no esenciales para Every Code.
  - Dependencia futura del historial como capability principal.

## Qué integrar de upstream primero
1. Prioridad alta: mejoras del stack LLM/chat que ya están en `upstream/main` y tocan la misma superficie que usa Every Code.
   - `thinking mode`
   - edición de `system prompt`
   - envío de mensajes con solo audio
   - copy/selective text
   - mejoras de input/config/max tokens/warnings
2. Prioridad media: refinamientos de Home/Settings/model import/model download que reduzcan drift visual y de flujo.
3. Prioridad media: cualquier ajuste requerido para que Tiny Garden quede exactamente en línea con upstream, ya que acaba de restaurarse y conviene congelar esa zona.
4. Prioridad baja: cambios upstream puramente internos que no mejoren UX ni reduzcan conflictos evidentes.

Rationale: el delta visible frente a upstream está muy concentrado en `ui/common/chat`, `ui/llmchat`, `ui/home`, `data/*` y strings. Es mejor traer primero lo que más valor aporta al uso diario y al mismo tiempo más se aleja con cada release.

## Recommended architecture direction
Separar el fork en dos capas:

- `upstream-compatible app layer`
  - navegación
  - chat UI
  - task modules
  - model/config flows
  - Tiny Garden y experiences
- `everycode-local layer`
  - service startup
  - localhost HTTP routing/contracts
  - prefs estrictamente de bridge
  - tests del bridge
  - opcionalmente un pequeño adapter hacia `LlmChatModelHelper`

La regla práctica debería ser: si una personalización local no es necesaria para que Every Code hable con el app, no debería vivir fuera de esa capa.

## Phased implementation plan
1. Baseline de integración y checkpoints.
   - Crear un branch de trabajo nuevo desde el estado actual sin limpiar el árbol existente.
   - Capturar un inventario de divergencia por áreas: `upstream/main..main`, commits del fork y archivos locales sin track.
   - Definir 2 checkpoints de rollback: `pre-history-removal`, `pre-upstream-chat-sync`.

2. Extraer el bridge HTTP como “overlay” del fork.
   - Mantener `LlmHttpService` y su arranque desde `MainActivity`, pero reducir sus dependencias directas sobre piezas cambiantes de chat UI.
   - Mover cualquier lógica de selección de modelo, serialización de requests/responses y toggles a clases/archivos claramente marcados como Every Code.
   - Objetivo: que una futura actualización de `ui/common/chat` o `ui/llmchat` no obligue a re-resolver el servidor HTTP manualmente.

3. De-integrar chat history antes de importar más upstream.
   - Cambiar la meta de “historial persistente” a “sesión efímera actual”.
   - Eliminar primero la navegación y entry points UI del historial:
     - `ROUTE_CHAT_HISTORY`
     - acceso desde `HomeScreen`
     - icono/acción de `CHAT_HISTORY`
     - `ui/history/*`
   - Luego desacoplar `LlmChatViewModel` de `ChatHistoryRepository` mediante una abstracción simple de session state efímero o directamente volver al flujo upstream si no requiere persistencia.
   - Finalmente retirar wiring de Room/SQLCipher/Hilt en `AppModule.kt` y limpiar `LlmHttpPrefs` del flag `history_enabled`.
   - Si se quiere compatibilidad transitoria, dejar import/export fuera de scope y no migrar datos; documentar que historial previo queda obsoleto.

4. Sincronizar primero el stack chat/UI con upstream.
   - Reaplicar sobre la base ya simplificada los commits/features recientes de upstream en:
     - `ui/common/chat`
     - `ui/llmchat`
     - `data/Config.kt`
     - `DataStoreRepository.kt`
     - `HomeScreen.kt`
     - `SettingsDialog.kt`
     - `strings.xml`
   - Hacerlo por bloques funcionales pequeños, no por mega-merge.
   - Recomiendo cherry-pick o patch-port por tema, no merge masivo, dado el árbol sucio y la personalización actual.

5. Revalidar y endurecer el HTTP API después del sync.
   - Confirmar compatibilidad de endpoints actuales:
     - `/ping`
     - `/generate`
     - `/v1/chat/completions`
     - `/v1/models`
     - `/v1/responses` si sigue siendo necesario para Every Code
   - Añadir validaciones explícitas para modelo no disponible, body oversize, timeouts, cancelación y errores de carga del modelo.
   - Mantener binding a loopback únicamente y documentar claramente el contrato estable del fork.

6. Integrar el resto de mejoras upstream de baja fricción.
   - Model import/model allowlist/download UX.
   - Tiny Garden solo para mantener paridad con upstream, no para extenderlo localmente.
   - Ajustes menores de config y strings.

7. Cerrar con saneamiento del fork.
   - Actualizar README para reflejar que el historial persistente ya no es prioridad o fue retirado.
   - Dejar una sección “fork-specific surface area” que enumere exactamente qué se conserva fuera de upstream.
   - Si hay tests del bridge o scripts de smoke test, documentarlos como gate previo a futuras sincronizaciones.

## Testing and validation
- Smoke funcional del fork:
  - app abre con upstream home/task flows intactos
  - Tiny Garden carga assets restaurados
  - Settings permite activar/desactivar bridge y cambiar puerto
  - servicio HTTP arranca y se detiene correctamente
- Smoke del API local:
  - `GET /ping`
  - `GET /v1/models`
  - request mínima a `/generate`
  - request tipo OpenAI-compatible a `/v1/chat/completions`
  - si aplica, `/v1/responses`
- Regression checks clave:
  - modelo por defecto y selección de modelo siguen funcionando
  - warm-up no rompe primer request
  - si se elimina historial, abrir/reabrir chats no intenta restaurar sesiones ni navegar a rutas antiguas
- Verificación manual Android:
  - foreground service notification
  - app relaunch con HTTP activado
  - cambio de modelo mientras Every Code usa el bridge
  - errores de modelo/archivo allowlist/importado

## Rollback checkpoints
- Checkpoint A: después de aislar HTTP y antes de retirar historial.
- Checkpoint B: después de retirar historial y antes de portar mejoras upstream de chat.
- Checkpoint C: después de cada bloque upstream importante, especialmente `ui/common/chat` y `ui/llmchat`.

Si un bloque rompe el bridge HTTP, volver al checkpoint anterior y re-portar solo el subconjunto funcional mínimo.

## Riesgos principales
- `LlmChatViewModel` y `LlmChatScreen` mezclan session lifecycle, UI state y persistencia; ahí es probable que aparezcan regresiones sutiles.
- `AppModule.kt` está cargando dependencias de Room/SQLCipher que upstream ya no necesita; si no se retiran pronto, seguirán produciendo conflictos en cada sync.
- `SettingsDialog.kt` hoy mezcla settings del app y settings del bridge; upstream seguirá tocando esa pantalla.
- `HomeScreen.kt` y `GalleryNavGraph.kt` son conflicto seguro por navegación e iconografía del historial.
- `LlmHttpService.kt` usa `Model`, allowlist y helpers LLM del app; cualquier cambio upstream en esas APIs puede romper el bridge si no se encapsula.

## Workflow de sync recomendado
- No trabajar con merges grandes desde `upstream/main` mientras el árbol siga sucio.
- Mantener una rama “fork-overlay” conceptualmente pequeña y documentada.
- Para cada release/oleada upstream:
  - revisar commits nuevos en `upstream/main`
  - agrupar por tema
  - portar primero cambios que no toquen la capa Every Code
  - re-ejecutar smoke tests del bridge antes del siguiente bloque
- Adoptar una política explícita:
  - upstream owns UI/UX general
  - fork owns localhost API contract
  - chat history persistente no se arrastra salvo que vuelva a ser un requisito real y justificable

## Open questions
- Every Code depende hoy de `/v1/responses` además de `/generate` y `/v1/chat/completions`, o puede reducirse el contrato soportado.
- Conviene dejar una mínima “sesión efímera” en memoria para UX local, o volver por completo al comportamiento upstream sin restore.
- El fork quiere seguir manteniendo redirects/HF token customizaciones propias, o también deberían acercarse a upstream si ya no aportan valor.

Mi recomendación concreta es: primero retirar o desactivar de forma estructural el chat history persistente, luego portar el stack de chat reciente de upstream, y solo después endurecer el bridge HTTP como capa estable del fork. Ese orden reduce muchísimo el conflicto futuro y deja a Every Code como una personalización pequeña y sostenible.
