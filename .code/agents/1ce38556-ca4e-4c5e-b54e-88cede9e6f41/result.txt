# Integration Plan: google-ai-edge/gallery Fork — Phased Implementation

## Situation Summary

**Fork state today:**
- Fork `main` has diverged from `upstream/main` by ~135 files, 10K insertions, 6K deletions.
- Fork additions cluster in three areas: HTTP API (`LlmHttpService`), chat history (`data/chathistory/`, `ui/history/`), and model allowlist changes.
- `upstream/main` is ahead on: thinking-mode support, config system enhancements, model allowlist updates (LiteRT-LM), TinyGarden (already restored in fork), memory optimizations.
- Chat history is confirmed no longer needed; it is the largest single source of merge complexity and maintenance debt.
- The HTTP API is the critical fork asset that must survive all merges.

---

## Phase 0 — Pre-flight Hygiene (1 day)

**Goal:** Create a stable, documented baseline before any destructive operations.

### 0.1 Commit the current dirty tree

The working tree is currently dirty with several modified files (`AGENTS.md`, `Config.kt`, `DataStoreRepository.kt`, `Model.kt`, `ModelAllowlist.kt`, `Tasks.kt`, `ModelImportDialog.kt`, `ModelManagerViewModel.kt`, `README.md`). These must be committed before any merge or rebase:

```
git add -p  # stage only intentional changes
git commit -m "chore: pre-integration checkpoint — dirty WIP"
```

**Risk:** Do not `git add .` — TinyGarden assets are already tracked; review each modified file before staging.

### 0.2 Tag the fork baseline

```
git tag fork-baseline-$(date +%Y%m%d)
```

This is your rollback anchor for the entire integration.

### 0.3 Refresh upstream

```
git fetch upstream
```

Confirm `upstream/main` HEAD SHA and log the divergence point:

```
git merge-base main upstream/main
```

The common ancestor is the point from which you cherry-pick or rebase. Record this SHA.

### 0.4 Document the fork diff explicitly

Run:
```
git diff $(git merge-base main upstream/main)..main --stat
```

Categorize every file into one of:
- **Fork-only** (new files upstream doesn't have): `LlmHttpService.kt`, `data/chathistory/*`, `ui/history/*`, AGENTS.md additions
- **Fork-modified** (both sides touched): `LlmChatViewModel.kt`, `AppModule.kt`, `GalleryNavGraph.kt`, `HomeScreen.kt`, `ModelAllowlist.kt`, `Tasks.kt`, `Config.kt`, `Model.kt`, `DataStoreRepository.kt`
- **Upstream-only** (new files in upstream we want): thinking-mode files, new config types, any new task screens

This list drives conflict triage in Phase 2.

## Phase 1 — Remove Chat History (1–2 days)

**Goal:** Excise the chat history subsystem before merging upstream. Doing this first shrinks the diff surface by hundreds of lines across the hotspot files, making the upstream merge significantly cleaner.

Do this on a dedicated branch `chore/remove-chat-history` off current `main`.

### 1.1 Delete the subsystem packages

These directories contain only fork-specific code with no upstream counterpart. Delete them entirely:

```
Android/src/app/src/main/java/com/google/ai/edge/gallery/data/chathistory/
Android/src/app/src/main/java/com/google/ai/edge/gallery/ui/history/
```

Files in scope:
- `ChatHistoryRepository.kt`, `ChatHistoryDatabase.kt`, `ChatHistoryDao.kt`
- `ChatHistoryEntities.kt`, `ChatHistoryKeyManager.kt`, `ChatCryptoManager.kt`
- `ChatAttachmentStorage.kt`, `ChatHistoryExport.kt`
- `ChatHistoryScreen.kt`, `ChatHistoryViewModel.kt`

### 1.2 Unwire `AppModule.kt`

Remove the six history-related `@Provides` functions:
- `provideChatCryptoManager`
- `provideChatHistoryKeyManager`
- `provideChatAttachmentStorage`
- `provideChatHistoryDatabase`
- `provideChatHistoryDao`
- `provideChatHistoryRepository`

Also remove the SQLCipher import and any `net.zetetic.android.database` references.

**Risk:** Hilt will fail the build if any remaining injected site still requests `ChatHistoryRepository`. Catch these at compile time; they are all in `LlmChatViewModel.kt` base class.

### 1.3 Unwire `LlmChatViewModel.kt` (highest risk file)

`LlmChatViewModelBase` takes `chatHistoryRepository: ChatHistoryRepository` as a constructor parameter. Remove it and replace the session/persistence logic with a no-op ephemeral approach:

- `initializeSession()` → just sets in-memory state, no DB lookup
- `persistMessages()` → delete entirely or make a no-op
- `reloadCurrentSessionMessages()` → no-op
- Remove `sessionId` parameter from navigation deep-link handling (or keep parameter but ignore it)

**What to keep:** All inference logic (`generateResponse`, `stopResponse`, `runAgain`), benchmark stats, and model lifecycle. Only the persistence layer is removed.

**Verification:** All three ViewModel subclasses (`LlmChatViewModel`, `LlmAskImageViewModel`, `LlmAskAudioViewModel`) must compile without history dependency.

### 1.4 Unwire `GalleryNavGraph.kt`

- Remove `navigateToChatHistory` callback from `GalleryNavHost` signature
- Remove the `chat_history` composable destination block
- Remove `sessionId` from the route pattern if no longer used (simplifies URL surface)
- Remove import of `ChatHistoryScreen`

### 1.5 Unwire `HomeScreen.kt`

- Remove the history icon button from the top app bar's secondary action
- Remove `navigateToChatHistory` parameter from the composable signature

### 1.6 Clean `LlmHttpPrefs` history toggle

`LlmHttpPrefs` has an `isHistoryEnabled()` flag that was bridged into `ChatHistoryRepository`. Since the repository is gone:
- Remove `isHistoryEnabled()` from `LlmHttpPrefs` if it controls only the now-deleted history path
- Keep all other prefs (port, payload logging)

**Risk:** `LlmHttpPrefs` may be referenced in Settings UI. Audit the settings screen composable before deleting any field.

### 1.7 Remove SQLCipher dependency

In `build.gradle.kts` (app module), remove:
```
implementation("net.zetetic:android-database-sqlcipher:...")
implementation("androidx.sqlite:sqlite-ktx:...")  // only if introduced for history
```

Also remove from `libs.versions.toml` if present.

### 1.8 Verification checkpoint

Build and run the app. All four task screens (Chat, Prompt Lab, Ask Image, Ask Audio) must function. No history icon in top bar. No DB initialization in logs.

**Rollback:** `git checkout fork-baseline-YYYYMMDD` and abandon the branch.

## Phase 2 — Upstream Feature Integration (2–4 days)

**Goal:** Bring in upstream features cleanly after the chat-history noise is gone.

Recommended strategy: **cherry-pick or selective merge**, not a full rebase. A full rebase rewrites all fork commits and makes the fork's history unreadable. Cherry-picking specific upstream feature commits is safer and auditable.

### 2.1 Categorize upstream commits

After `git fetch upstream`, run:
```
git log --oneline $(git merge-base main upstream/main)..upstream/main
```

Cluster commits into buckets:
- **Model allowlist / LiteRT-LM**: Low conflict risk (the fork already made allowlist changes, so merge will require manual reconciliation of `ModelAllowlist.kt` and `model_allowlist.json`)
- **Thinking-mode UI and inference**: Medium risk — involves `LlmChatViewModel` which we just simplified; upstream changes here should be clean to apply
- **Config system enhancements** (`Config.kt`, `Model.kt`): Low–medium risk; fork added no structural changes here
- **TinyGarden improvements**: Low risk — TinyGarden assets are already restored; upstream may have additional JS updates
- **Memory optimization / build changes**: Low risk
- **Any new task screens** (e.g., new generative AI experiences): Low risk — new files

### 2.2 Integration order (sequential)

**Step 1 — Build system and dependencies first**

Cherry-pick upstream build changes (`build.gradle`, `libs.versions.toml`, Gradle wrapper). These should be nearly conflict-free and establish the right dependency baseline for everything that follows.

**Step 2 — Data layer** (`Config.kt`, `Model.kt`, `ModelAllowlist.kt`, `DataStoreRepository.kt`)

The fork's allowlist changes (LiteRT-LM model names) must be preserved. Strategy:
1. Apply upstream changes to these files.
2. Then re-apply the fork's model entries on top.
3. Key conflict zone: `model_allowlist.json` — the fork may have different model entries than upstream. Manually merge: keep all upstream models, keep any fork-specific models (if any were added beyond allowlist renaming), and resolve any duplicate name collisions.

**Step 3 — Tasks.kt**

The fork may have added `LLM_TINY_GARDEN` support or modified task configs. Upstream likely has the same. Manual merge: ensure TinyGarden task is present exactly once with correct config. Check `llmSupportTinyGarden` flag on `AllowedModel`.

**Step 4 — Thinking mode**

This is likely the largest new upstream feature. Involves:
- New config keys or a new `Config` subtype
- Changes to `LlmChatViewModel` inference path
- Possibly a new UI composable for thinking output
- Changes to `LlmChatModelHelper` (not fork-modified, so apply cleanly)

Because we stripped `LlmChatViewModel` of history logic in Phase 1, applying upstream's thinking-mode changes should touch a cleaner base. Apply these carefully — the inference loop (`generateResponse`) is shared with `LlmHttpService`'s model helper.

**Critical:** After applying thinking-mode changes, verify that `LlmHttpService` still calls `LlmChatModelHelper` with a compatible API. If `LlmChatModelHelper` gained new required parameters for thinking mode, update the HTTP service's model invocation accordingly (likely just passing `null` or a default for thinking-mode parameters when called from the HTTP path).

**Step 5 — UI screens**

`HomeScreen.kt`, `GalleryNavGraph.kt`, and task screen composables. These are moderate conflict risk because:
- Phase 1 removed history navigation from both files
- Upstream may have added new navigation entries or new task card styles

Apply upstream UI changes, then verify the Phase 1 removals are still clean (history routes not re-introduced).

**Step 6 — New task screens (if any)**

New composable files upstream added (e.g., for any new generative AI experience): these should be conflict-free new file additions.

### 2.3 Hotspot file conflict strategy

| File | Conflict risk | Strategy |
|------|--------------|----------|
| `LlmChatViewModel.kt` | High | Phase 1 cleans it first; apply upstream changes to the simplified base |
| `AppModule.kt` | High | Phase 1 removes history DI; upstream adds DI for new features; apply cleanly |
| `GalleryNavGraph.kt` | Medium | Phase 1 removes history route; upstream adds new routes; apply cleanly |
| `HomeScreen.kt` | Medium | Phase 1 removes history button; upstream may restyle cards; apply cleanly |
| `ModelAllowlist.kt` | Medium | Manual merge of model lists |
| `Tasks.kt` | Low | Additive |
| `Config.kt` | Low | Additive (new config types) |
| `LlmHttpService.kt` | **None** | Fork-only file, upstream never touches it |

### 2.4 Verification checkpoint

After each upstream step:
1. `./gradlew assembleDebug` — must pass
2. Run on device: all pre-existing tasks (Chat, Prompt Lab, Image, Audio) must work
3. HTTP API must still respond on port 9006 (test with `curl 127.0.0.1:9006/ping`)
4. TinyGarden must load

**Rollback:** Branch `chore/remove-chat-history` (from Phase 1) is your rollback point. Each step above should be its own commit so individual cherry-picks can be reverted.

## Phase 3 — Harden the HTTP API (1 day)

**Goal:** Ensure `LlmHttpService` is robust against upstream model-layer changes.

### 3.1 Audit `LlmChatModelHelper` interface

`LlmHttpService` invokes the model helper directly. After upstream model changes land:
- Check `LlmChatModelHelper`'s function signatures haven't changed in ways that break the HTTP service
- Check that the model warm-up sequence still works (the service pre-initializes the model on start)
- Check that `generateAsync` / streaming callbacks are still compatible with the HTTP service's SSE emission loop

### 3.2 Reconcile model selection logic

The HTTP service has hardcoded preference for `"Gemma3-1B-IT"`. After upstream allowlist updates:
- Verify this model name still exists in the updated allowlist, or update the preferred model name to match whatever LiteRT-LM variant is the default
- The import directory path (`__imports`) should be unchanged

### 3.3 Add a smoke-test script

Create a shell script (not part of Android source, just in the repo root) for manual verification:

```bash
#!/bin/bash
# test_http_api.sh
BASE="http://127.0.0.1:9006"
curl -s $BASE/ping
curl -s $BASE/v1/models
curl -s -X POST $BASE/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"say hello"}]}'
```

This makes regression testing the HTTP API mechanical rather than manual.

### 3.4 Consider extracting HTTP API wiring into a companion interface

**Optional, low priority.** Right now `LlmHttpService` reaches directly into `LlmChatModelHelper`. If upstream refactors `LlmChatModelHelper` significantly (e.g., renames methods for thinking-mode), the service breaks silently at runtime.

Mitigation: define a thin `LlmInferenceBackend` interface (3–4 methods: `init`, `generate`, `generateStreaming`, `cancel`) that `LlmHttpService` calls, implemented by a small adapter that wraps `LlmChatModelHelper`. This insulates the HTTP service from upstream model-helper churn. Do this only if you have capacity — it adds a layer but dramatically reduces future breakage risk.

## Phase 4 — Workflow / Sync Strategy for Future Drift Reduction

**Goal:** Make future upstream integrations take hours, not days.

### 4.1 Keep fork additions in dedicated files as much as possible

The worst merge conflicts happen in shared files that both fork and upstream modify. The HTTP API is already well-isolated in `LlmHttpService.kt` (a file upstream never touches). Continue this pattern:
- Any new fork-specific settings: add to `LlmHttpPrefs.kt` rather than modifying `DataStoreRepository.kt`
- Any new fork-specific DI bindings: consider a separate `HttpApiModule.kt` instead of adding to `AppModule.kt`
- Any new fork-specific navigation: prefer a composable extension function over modifying `GalleryNavGraph.kt` inline

### 4.2 Minimize modifications to upstream files

Current upstream-file modifications in the fork:
- `LlmChatViewModel.kt` (history wiring, now removed in Phase 1 → should be clean)
- `AppModule.kt` (history DI, removed → should be clean)
- `GalleryNavGraph.kt` (history route, removed → should be clean)
- `HomeScreen.kt` (history button, removed → should be clean)
- `ModelAllowlist.kt` / `Tasks.kt` / `Config.kt` / `Model.kt` (additive data changes)

After Phase 1 and 2, the only remaining upstream-file modifications should be the model allowlist changes. These are inherently mergeable (additive list entries).

### 4.3 Adopt a "sync branch" workflow

Instead of merging directly to `main`:

```
upstream/main
    ↓  (periodic cherry-pick or merge)
sync/upstream-YYYYMMDD   ← integration branch
    ↓  (merge after verification)
main
```

This keeps `main` always in a verified buildable state and lets integration work happen in isolation.

### 4.4 Track upstream at the commit level

Maintain a file `UPSTREAM.md` (or a line in `AGENTS.md`) recording:
- The upstream SHA that was last integrated
- The date
- A brief note on what was pulled in

This prevents "what have we already merged?" confusion in future sessions.

Example entry:
```
## Upstream Sync Log
- 2026-03-28: Integrated up to upstream SHA abc1234 (thinking mode, LiteRT-LM allowlist)
```

### 4.5 Run upstream integration on a schedule

Given the pace of upstream development, aim for one integration cycle per 4–6 weeks. Letting it drift for months compounds the conflict surface exponentially. A regular shallow pull (2–5 upstream commits at a time) is much cheaper than a large rebase every quarter.
